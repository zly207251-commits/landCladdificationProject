# Comparison of a Specific Area Over Time

## الهدف
إنشاء وثيقة مستقلة تشرح طريقة مقارنة منطقة محددة في صورة واحدة عبر تواريخ مختلفة، باستخدام بيانات بلاطات جاهزة بدل تحميل خريطة العالم كاملة.

## الفكرة الأساسية
- نحدد منطقة جغرافية معينة بواسطة إحداثيات أو حدود `bbox`.
- نستخدم خادم بلاطات أو API صور تاريخية ليقدم لنا الصور لتلك المنطقة فقط.
- نحصل على صورتين أو أكثر لنفس المنطقة في تواريخ مختلفة.
- نقارن بين الصور لاكتشاف التغييرات والإنشاءات الجديدة.

## لماذا هذه الطريقة أفضل
- لا نحتاج إلى تحميل صورة العالم بكاملها.
- نقل البيانات أصغر وأسرع.
- لا نعتمد على تخزين ضخم أو معالجة صورة كبيرة جداً.
- التركيز يكون على المنطقة المهمة فقط.

## خطوات التنفيذ

### 1. تحديد المنطقة المطلوبة
- يجب تحديد المنطقة عبر:
  - `latitude/longitude` مركز المنطقة + نصف القطر
  - أو مربع حدود `bbox` ([minLon, minLat, maxLon, maxLat])
  - أو مضلع `polygon` إذا كانت المنطقة غير مستطيلة.

### 2. اختيار مصدر بيانات مناسب
- استخدم مصدر يدعم التاريخ والزمن، مثل:
  - Google Earth Engine (للتحليل التاريخي)
  - Copernicus Sentinel
  - USGS Landsat
  - NASA GIBS إذا كان العرض الزمني متاحاً
- إذا لم يكن المصدر الرسمي متاحاً، يمكن استخدام `tiles` جاهزة لمنطقة محددة.

### 3. طلب الصور للمنطقة والتواريخ
- اطلب صورة أو بلاطات فقط للمنطقة المحددة بدلاً من العالم كله.
- لكل تاريخ تحتاج:
  - صورة قديمة
  - صورة لاحقة
- إذا كان API يدعم التاريخ مباشرة، استخدم معلمات التاريخ في الطلب.

### 4. تخزين الصور أو نتائج البلاطات
- خزّن الصور أو البلاطات التي تم جلبها في الـ backend.
- يمكن حفظ كل تاريخ كطبقة منفصلة أو كمسار ملفات منفصل.
- الأفضل أن يكون التخزين على الشكل التالي:
  - `history/{task_id}/{date}/tiles/...`
  - أو `history/{task_id}/{date}/image.png`

### 5. معالجة المقارنة
- يمكن إجراء المقارنة بعدة طرق:
  - مقارنة بكسل إلى بكسل إذا كانتا بنفس الدقة والمحاذاة.
  - مقارنة طبقات حدود جديدة مقابل قديمة.
  - استخراج اختلافات في المباني أو الغطاء الأرضي.
- إذا كانت المقارنة تستهدف اكتشاف "ما تم استحداثه" فالأفضل استخدام:
  - تحليل الفرق `change detection`
  - مقارنة حدود المباني/الطرق

### 6. عرض النتائج في الواجهة
- عرض طبقة تاريخ قديم وطبقة تاريخ جديد.
- استخدام شريط تمرير زمني `time slider` للمقارنة البصرية.
- أو عرض الفرق في منطقة محددة على شكل لون أو تمييز.

## تكامل مع نظامنا الحالي
- في `backend`:
  - أضف واجهة جديدة لجلب الصور التاريخية أو استدعاء API التاريخي.
  - خزّن النتائج ضمن مهمة مقارنة زمنية.
- في `frontend`:
  - أضف صفحة جديدة أو مكون جديد يعرض مقارنة المنطقة.
  - استخدم علامة تحديد المنطقة و`task_id` أو `date_range`.
  - اعرض الصور كمصدرين متزامنين أو كطبقتين قابلتين للتبديل.

## المتطلبات الأساسية
- إمكانية تحديد الإحداثيات بدقة.
- مصدر بيانات يدعم التاريخ.
- عملية تحميل لبلاطات المنطقة فقط.
- آلية لحفظ البيانات التاريخية في الخادم.
- طريقة لعرض الطبقات المقارنة في الواجهة.

## مثال مبسط للعملية
1. المستخدم يحدد المنطقة.
2. ينفذ النظام طلب API إلى مصدر الصور للتاريخ الأول.
3. ينفذ النظام طلب API للتاريخ الثاني.
4. يخزن الصور أو البلاطات.
5. يعرض واجهة مقارنة بين التاريخين.

## نقاط يجب الانتباه لها
- يجب أن تكون الصور من نفس المنطقة والدقة نفسها قدر الإمكان.
- بعض مصادر الصور التاريخية قد تحتاج تسجيل أو مفاتيح API.
- بعض الخدمات المجانية لها حدود استخدام `quota`.
- قد تحتاج معالجة إضافية لمواءمة الصور إذا لم تكن متمركزة بنفس الإحداثيات بدقة.

## الخلاصة
الخطة تعتمد على جلب بيانات المنطقة المطلوبة فقط، ثم مقارنة تواريخ مختلفة في تلك المنطقة. هذا يوفر أداء وبيانات أقل ويسمح باكتشاف التغييرات بدقة أكبر بدون تحميل خريطة العالم كاملة.
